## 8. 실습 목표 (2) : scene tree 구조 변경, 1인칭으로 수정

이제 최신 three js 를 적용했으니, 본격적으로 코드 수정을 해보겠습니다.

간단하게, space bar 를 누르면 기존 3인칭 카메라 시점을 비행기 1인칭 시점으로 변환하도록 해보겠습니다.

그 전에, 현재 게임 씬을 분석해보겠습니다.

먼저 scene 의 좌표계를 시각화해서 이 게임의 x y z 축이 어느 방향인지 확인해봅시다.

- "좌표계 축이 보이게 할 수 있어? 월드 기준 x,y,z 가 어느 방향인지 보고싶어."

피피티 왼쪽 아래 그림처럼, 빨간색이 x, 초록색이 y, 파란색이 z 축입니다.

3인칭 카메라가 보는 방향이 -z 축, 비행기가 보는 방향이 +x 축, 위 방향이 +y 축인 오른손 좌표계네요.

추가로 조금더 자유로운 시점 전환을 추가해서, 실제로 씬이 어떻게 구성되어있는것인지도 확인해보겠습니다. 

three js 에는 orbit control 이라는 기능이 있어서, 이걸 적용하면 마우스를 이용해서 자유롭게 시점을 이동할 수 있습니다.

- "키보드 spacebar를 눌러서 orbit control 로 자유로운 시점 전환을 토글하게 수정할 수 있어?"

먼 거리에서 씬을 보니 오브젝트들이 흐려지네요. 아마도 fog 기능이 켜져있거나, camera farplane 의 거리가 짧아서 culling 이 되고있을 가능성도 있습니다. 정확히 확인해보겠습니다.

- "멀리서 씬을 보니까 일정 거리 바깥은 안보여. fog 또는 farplane 설정이 있는지 확인해줘"

이제 먼거리의 오브젝트들이 잘보이네요.

확인해보니, 월드 기준에서 비행기 위치는 가만히 있고, 바다 메시와 여러 오브젝트들이 바다 메시 중심을 축으로 반대로 회전하고 있습니다.

이제 비행기의 1인칭 카메라를 구현해보기 전에 마지막으로, 현재 scene tree 구조를 확인해보겠습니다. agent 가 코드 수정 말고도 문서 작성과 자료 정리도 잘합니다. scene tree 를 다이어그램으로 그려달라고 해보겠습니다.

- "카메라를 포함한 scene tree 분석해서 다이어그램 파일로 만들어줘."

(Markdown Preview Mermaid Support 익스텐션 설치해야 보임)

간단히 ppt 의 왼쪽 그림으로 정리할 수 있습니다.

지금 카메라의 움직임을 보시면, 마우스 상하 이동에 어느정도 따라서 움직이고 있습니다. 

이제 spacerbar 를 누르면 카메라를 비행기 1인칭 시점으로 바꾸도록 해보겠습니다.

- "spacebar 로 시점 전환하는거에 카메라가 기존 3인칭에서 비행기에 자식으로 붙는 1인칭 시점을 추가하고 싶어. 계획을 세워서 보여줘."

- "계획대로 실행해줘"

보시면 비행기의 회전이 카메라에도 똑같이 적용되고있습니다. 아까 영상처럼 비행기의 translation 만 따라가고 rotation은 영향을 받지 않도록 해보겠습니다. scene tree 개념을 응용해서, 비행기를 새로운 오브젝트(airplane_box)의 자식으로 붙이고, 그 오브젝트에 기존 비행기의 translation 을 적용하면 되겠죠?
그러면 camera 와는 관계없이 airplane 의 rotation 이 적용됩니다.

- "1인칭 시점에서, 비행기의 회전은 카메라에 영향을 안받게 할 수 있어? 비행기의 translation 만 따라가고 rotation은 적용안되게. 비행기 메시에 새로운 부모 오브젝트를 추가하고, 해당 오브젝트에 기존 비행기에 적용하던 translation 를 적용하고, 카메라를 해당 오브젝트의 자식으로 설정하면 될거같아."

## 9. 실습 목표 (3) : 게임 로직 바꿔보기

이제 간단하게 게임 로직도 수정해보겠습니다.

방금전에 1인칭 기능을 추가했으니, 1인칭에 맞는 조작법을 추가해보겠습니다.

기존 마우스 위아래 이동 대신, w s로 상하 이동, a d 로 좌우 이동 하도록 구현해봅시다. 

아까 확인한 좌표계 축 방향대로, w s 키를 누르면 y 축, a d 키는 z 축 방향으로 움직이게 하면 될거같습니다.

- "기존 마우스 위아래 이동을 제거하고, 키보드 w s 로 상하(y축), a d 로 좌우(z축)로 비행기가 자유롭게 움직이게 구현할 수 있어? 축은 월드기준이야."

## 10. 실습 목표 (4) : gpu vertex processing

본격적인 셰이더 코딩에 들어가기 전에, 바다 메시의 파도가 어떻게 실시간으로 움직이면서 파도가 치는걸 표현하는지 알아보겠습니다.

바다가 계속 빠르게 회전하고 있어서 각각의 점들이 어떻게 움직이는지 잘 안보이니까 속도를 줄여보겠습니다.

"sea mesh 천천히 회전하게 하려면 어떤 값을 수정해야해?"

바다 메시의 각각의 버텍스들이 하나하나 움직이고 있습니다. 어떻게 구현했는지 확인해보겠습니다.

"sea mesh 의 vertex 가 파도치는 코드 어떻게 구현되고 있는지 분석해줘."

기존 구조는 cpu가 바다 geometry의 vertex 들을 for 문을 돌면서 하나하나 수정하는 방식입니다.

ppt 사진의 왼쪽같은 경우죠.

cpu와 달리 gpu 는 같은 연산을 동시에 많이 실행하는 병렬처리에 강합니다.

기존 코드를 gpu 를 이용해서 효율적인 병렬처리가 가능하도록 수정해보겠습니다.

- "저걸 gpu에서 처리하도록 바꿀수 있어? 지금은 cpu 에서 몇백개의 vertex 를 매프레임마다 비효율적으로 수정하는거 아니야? gpu 로 병렬 처리하도록 수정하는 계획 작성해줘."

## 11. 실습 목표 (5) : Advanced shader

이제 마지막으로, 바다 메시에 특별한 셰이더 코드를 적용해보겠습니다.

https://threejs.org/examples/

이 페이지에는 three js 관련 예제들을 확인할 수 있습니다.

여기에서 저는 lava 를 검색해서 용암 느낌의 셰이더가 있는지 찾아보겠습니다.

이 링크를 복사해서 antigravity 에게 적용해달라고 해보겠습니다.

- "https://threejs.org/examples/webgl_shader_lava.html  이거를 지금 sea mesh 에 적용시켜줄 수 있어?"

용암이 흐르는 속도가 너무 빠르네요. 셰이더에 전달되는 시간 값에 추가적인 처리가 필요해보입니다. 근데 이 시간 값은 아까 수정한 vertex 가 파도처럼 움직이는 효과에도 사용되고 있으니까, 잘못해서 건드리지 않도록 작업을 해보겠습니다.

- "용암 속도가 너무 빨라. 기존 버텍스 변형 속도에는 영향 안가게 용암 흐르는 속도를 낮춰줄 수 있어?"

셰이더 적용이 잘 되었습니다. 아직 부족한 느낌이 드는데, 이건 bloom 이라는 postprocessing을 적용해서 해결 가능합니다.

- "sea 메시의 용암에 bloom 효과도 적용 가능해?"
